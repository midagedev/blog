해당 글은 "More What, Less How"라는 제목으로, 프로그래밍 언어와 기술의 진화에 대한 이야기를 담고 있습니다.

1975년, 저자의 아버지는 어셈블리 언어 프로그래머로서, C와 같은 고급 언어에 대한 이야기를 점점 더 많이 듣게 되었습니다. C 언어는 프로그래머가 프로그램이 무엇을 해야 하는지 결정하고, C 컴파일러가 어떻게 그것을 수행할지 결정하도록 해주는 약속이었습니다.

하지만, 컴파일러는 수동으로 코드를 작성한 어셈블리 언어만큼 효율적이지 않을 것이며, 컴파일러는 코드를 작은 공간에 집어넣는 데 있어 어셈블리 언어만큼 능력이 없을 것이라는 우려가 있었습니다. 또한, 컴파일러가 실행하는 데 걸리는 시간이 어셈블러보다 길다는 문제도 있었습니다. 그러나 시간이 지남에 따라 컴파일러는 점점 더 나아졌습니다.

이러한 변화는 프로그래머가 프로그램이 무엇을 해야 하는지 말하게 하고, 모델이 그 작업을 어떻게 수행할지 파악하도록 하는 대형 언어 모델을 약속하는 것과 비슷합니다.

저자는 자신의 기술의 90%는 이제 가치가 없지만, 남은 10%는 이전보다 1000배 가치가 있다고 트윗했습니다. 이 90%와 10%가 무엇인지는 확실하지 않지만, 분석적 기술에서 찾아볼 수 있다고 말합니다. 즉, 무엇을 선택할 것인지, 피드백에 따라 무엇을 변경할 것인지, 이상한 비유를 찾아내는 것, 아무도 연결해볼 생각을 하지 않은 이상한 것들을 연결하는 것 등이 중요하다는 것입니다.

이 새로운 세상에서 처음부터 해결책을 창출하는 합성적인 기술은 가치가 떨어지지만, 문제와 해결책을 분석하는 기술은 가치가 크게 상승했습니다.

저자의 아버지는 어셈블리 언어에서 C 언어로 넘어갔고, 그는 여전히 같은 종류의 마이크로 컨트롤러 코드 작업을 수행했지만, 컴파일러는 그가 더 효율적으로 작업할 수 있게 해주었습니다. 그는 '어떻게'하는 기술을 포기하고 '무엇'에 더 집중하도록 배웠습니다.

이 이야기의 아이러니한 결말은, 저자가 대학을 졸업하고 객체 지향 프로그래밍이 대세가 되었을 때, 그의 아버지는 객체 지향 프로그래밍을 이해하려고 시도했지만 실패했다는 것입니다. 그는 마이크로 컨트롤러를 C로 계속 작성했고, 결국 수요가 사라지고 그는 은퇴하게 되었습니다.

  1. 모델 타입: ARIMA는 통계 기반의 선형 모델입니다. 시계열의 패턴을 인식하고 이를 사용하여 미래 값을 예측하는데, 자기 상관성이 있는 선형 시계열 데이터에 적합합니다. 반면에 LSTM은 인공신경망의 한 종류로, 비선형 패턴을 학습하고 복잡한 시계열 데이터에서 더욱 효과적으로 작동합니다.
  2. 시간 의존성: ARIMA는 고정된 시간 창(window) 내에서만 의존성을 인식할 수 있습니다. 즉, 창의 크기가 너무 커지면 모델의 성능이 저하될 수 있습니다. 반면에 LSTM은 "장기 의존성" 문제를 해결하기 위해 설계되었으며, 장기적인 의존성을 가진 데이터를 처리하는 데 더욱 유리합니다.
  3. 데이터 전처리: ARIMA는 정상 상태의 데이터를 필요로 합니다. 즉, 데이터가 일정한 평균과 분산을 가져야 합니다. 이를 위해 종종 차분(differencing)이나 변환(transformation)과 같은 전처리 단계가 필요합니다. LSTM은 이러한 전처리 과정이 필수적이지는 않지만, 종종 데이터의 스케일링이 필요할 수 있습니다.
  4. 학습 시간: LSTM은 일반적으로 ARIMA보다 학습 시간이 더 길다. LSTM은 비선형적인 특징을 학습하고, 이를 위해 더 많은 파라미터를 학습해야 하기 때문입니다.
  5. 해석 가능성: ARIMA 모델의 장점 중 하나는 그 결과를 해석하는 데 있어서의 명확성입니다. ARIMA 모델의 계수는 통계적으로 의미를 가질 수 있습니다. 반면에 LSTM은 일반적으로 '블랙박스' 모델로 여겨지며, 이는 그들의 내부 작동 방식을 이해하거나 설명하는 것이 어렵다는 의미입니다.

이러한 차이점에도 불구하고, ARIMA와 LSTM 모두 시계열 데이터를 다루는 데 유용한 도구입니다. 어떤 알고리즘이 가장 적합한지는 문제의 특성, 사용 가능한 데이터, 그리고 특정 작업에 대한 요구 사항에 따라 달라집니다.

  • create는 POST 요청이 들어왔을 때 데이터를 생성하는 뷰의 로직입니다. request 객체를 받아 해당 데이터를 이용해 새로운 인스턴스를 생성하고, 응답을 리턴하는 로직을 포함합니다. create 메서드는 API 요청에 대한 전체적인 흐름을 담당합니다.
  • 반면, perform_create는 create가 진행되는 과정 중, 실제로 새로운 데이터를 데이터베이스에 저장하는 역할을 합니다. perform_create 메서드에는 직접적으로 request 객체를 넘겨받지 않고, create로부터 전달받은 serializer와 validated_data를 이용해 데이터를 저장합니다.

즉, create 메서드가 전체적인 생성 흐름을 담당하는 반면, perform_create는 그 과정 중 데이터 저장에 집중하는 역할을 합니다.

이러한 구조는 코드의 모듈화와 재사용성을 향상시키며, 필요에 따라 perform_create를 오버라이드하면 create 메서드의 흐름은 그대로 유지하면서 데이터 저장 방식만을 변경하는 등의 유연한 커스터마이징을 가능하게 합니다.

from rest_framework import viewsets, status
from rest_framework.response import Response
from .serializers import YourModelSerializer
from .models import YourModel

class YourModelViewSet(viewsets.ModelViewSet):
    queryset = YourModel.objects.all()
    serializer_class = YourModelSerializer

    def create(self, request, *args, **kwargs):
        serializer = self.get_serializer(data=request.data)
        if serializer.is_valid(raise_exception=True):
            self.perform_create(serializer)
            return Response(serializer.data, status=status.HTTP_201_CREATED)

    def perform_create(self, serializer):
        # 여기에서 실제로 모델 인스턴스를 생성합니다.
        serializer.save()

위 코드에서는 create 메서드에서 먼저 요청된 데이터(request.data)를 serializer를 통해 검증(is_valid())합니다. 만약 검증이 성공하면, perform_create 메서드를 통해 데이터를 실제로 저장하고, 성공적으로 생성되었다는 응답을 보냅니다.

perform_create는 serializer의 save 메서드를 호출하여 실제 데이터를 데이터베이스에 저장합니다. 필요에 따라 이 부분을 오버라이드하여 특정 로직을 추가하거나 변경할 수 있습니다.

"CommonJS is not going away"라는 제목의 기사는 CommonJS가 여전히 유지될 것이며, 이는 괜찮은 일이라고 말하고 있습니다. CommonJS는 2009년에 개발되어 Node.js에 의해 널리 사용되는 모듈 형식입니다. 반면 ECMAScript 모듈(ES 모듈)은 2015년에 도입되었습니다. ES 모듈은 브라우저에서 작동하도록 설계되었으며, 파일이 네트워크를 통해 로드된다는 점에서 비동기적입니다.

CommonJS의 장점 중 하나는 더 빠른 시작 시간입니다. 큰 애플리케이션의 경우 ES 모듈보다 CommonJS가 더 빠르게 시작됩니다. 또한 CommonJS는 동적 모듈 로딩을 허용합니다. 이는 플러그인 시스템이나 사용자 상호작용에 기반한 특정 구성 요소의 지연 로딩과 같은 시나리오에서 유용할 수 있습니다.

또한 npm에 이미 수백만 개의 CommonJS 모듈이 게시되어 있으며, 이 중 많은 모듈은 더 이상 활발히 유지되지 않지만 기존 프로젝트에 매우 중요합니다. 따라서 모든 패키지가 ES 모듈을 사용하도록 기대할 수는 없습니다.

Bun은 최근 버전에서 CommonJS를 기본적으로 구현하고 있으며, CommonJS 파일을 특별한 "동기식 ESM" 형식으로 변환하고 있습니다. 또한, Bun은 최근에 __esModule 주석을 지원하는 기능을 추가했습니다. 이는 CommonJS 모듈이 exports.default를 기본 내보내기로 해석해야 함을 나타내는 비공식적인 방법입니다.

결론적으로, CommonJS는 이미 여기에 있으며 존재하는 이유가 있습니다. Bun은 CommonJS를 중요하게 생각하고 있으며, 이를 현대적인 방식으로 지원하고 있습니다.

"CommonJS is hurting JavaScript"라는 제목의 기사는 JavaScript의 발전을 저해하는 요인으로 CommonJS 모듈 시스템을 지목하고 있습니다. 요약하면 다음과 같습니다.

  • JavaScript는 웹 개발에서 널리 사용되는 언어로, 그러나 과거의 유산인 CommonJS로 인해 발전이 저해되고 있다.
  • CommonJS는 약 15년 전에 JavaScript가 브라우저를 넘어 서버 측으로 확장되면서 등장했다. 당시 JavaScript는 소스 코드를 더 효율적으로 관리할 필요가 있었고, 이를 위해 모듈화가 필요했다.
  • CommonJS는 Node.js의 모듈 시스템으로 채택되었으며, require(), module.exports 등의 문법을 제공했다. 이는 서버 측 코드와 클라이언트 측 코드의 요구 사항이 다르다는 것을 인식한 결과였다.
  • 그러나 CommonJS에는 몇 가지 문제점이 있었다. 모듈 로딩이 동기식이었고, 트리 쉐이킹(tree shaking)이 어려워 불필요한 모듈을 제거하는데 한계가 있었다. 또한 브라우저와 네이티브로 호환되지 않아 번들러와 트랜스파일러가 필요했다.
  • 2013년경, TC39 위원회는 CommonJS 모듈의 후속 버전으로 ECMAScript 모듈(ESM)을 개발하기 시작했다. ESM은 웹을 위해 최적화되어 있으며, 비동기 모듈 로딩, 브라우저 호환성, 정적 분석, 트리 쉐이킹 등을 지원한다.
  • Node.js는 CommonJS와 ESM을 모두 지원하기로 결정했으나, 두 시스템 간의 상호 운용성에 대한 명확한 합의에는 도달하지 못했다.
  • 기사는 2023년 현재, CommonJS를 지원하는 것이 더 이상 무시할 수 없는 문제가 되었으며, ESM으로 완전히 전환할 시점이라고 주장한다. ESM은 개발자에게 더 나은 솔루션을 제공하며, 사용자에게도 더 나은 경험을 제공한다.

commonjs와 esm의 비교

CommonJS와 ECMAScript Modules(ESM)은 JavaScript 모듈 시스템의 두 가지 주요 형태입니다. 각각의 특징과 차이점은 다음과 같습니다.

정의 방식:

  • CommonJS: module.exports를 사용하여 모듈을 내보내고, require() 함수를 사용하여 모듈을 가져옵니다.
// math.js
module.exports = {
  add: (a, b) => a + b,
  subtract: (a, b) => a - b
};

// app.js
const math = require('./math.js');
  • ESM: export 키워드를 사용하여 모듈을 내보내고, import 키워드를 사용하여 모듈을 가져옵니다.
// math.js
export const add = (a, b) => a + b;
export const subtract = (a, b) => a - b;

// app.js
import { add, subtract } from './math.js';

로딩 메커니즘:

  • CommonJS: 동기적으로 모듈을 로드합니다. 이는 서버 환경에서 적합하지만, 브라우저에서는 성능 문제를 일으킬 수 있습니다.
  • ESM: 비동기적으로 모듈을 로드합니다. 이는 브라우저 환경에서 더 효율적입니다.

호환성:

  • CommonJS: Node.js에서 널리 사용되며, 브라우저에서는 번들러(예: webpack)를 사용하여 변환해야 합니다.
  • ESM: 최신 브라우저에서 직접 지원되며, Node.js에서도 지원됩니다.

정적 분석과 최적화:

  • CommonJS: 런타임에 모듈 구조를 해석하기 때문에 정적 분석이 어렵고, 트리 쉐이킹과 같은 최적화 기법을 적용하기 어렵습니다.
  • ESM: 모듈 구조가 정적이므로 빌드 타임에 분석이 가능하고, 트리 쉐이킹과 같은 최적화 기법을 적용할 수 있습니다.

상호 운용성:

  • CommonJS: ESM과 직접적으로 호환되지 않으며, 변환 단계가 필요합니다.
  • ESM: 최신 Node.js 버전에서는 CommonJS 모듈을 가져올 수 있으나, 일부 제약 사항이 존재합니다.

커뮤니티와 채택:

  • CommonJS: 과거에 널리 사용되었으나, 현재는 점차 ESM으로 이동하는 추세입니다.
  • ESM: 현대적인 JavaScript 개발에 권장되며, 커뮤니티와 새로운 라이브러리에서 널리 채택되고 있습니다.

결론적으로, ESM은 브라우저와 서버 모두에서 사용하기에 더 적합하며, 최적화 기법과 더 나은 문법을 제공합니다. CommonJS는 여전히 많은 레거시 프로젝트와 라이브러리에서 사용되고 있지만, 새로운 프로젝트에서는 ESM을 사용하는 것이 좋습니다.

GIL이란?

Python의 GIL이란 Global Interpreter Lock의 약

]]>https://blog.midagedev.com/python-no-gil/64b157d3b104a3f8363cfa92Fri, 14 Jul 2023 14:26:32 GMTIf PEP 703 is accepted, Meta can commit three engineer-years to no-GIL CPython
아래의 내용은 위 글을 보고 chatgpt와의 대화를 정리한 것입니다.

GIL이란?

Python의 GIL이란 Global Interpreter Lock의 약자로, 한 번에 하나의 스레드만 Python 바이트코드를 실행하도록 제한하는 메커니즘이다.

이는 Python 인터프리터가 멀티스레드 환경에서 동시에 실행되는 스레드 간에 데이터를 안전하게 처리하도록 도와준다. 그러나 이로 인해 멀티코어 프로세서에서 Python의 멀티스레딩이 효과적으로 활용되지 못하는 단점이 있다.
GIL은 CPython에서만 발견되며, Jython이나 IronPython과 같은 다른 Python 구현에서는 이러한 제한이 없다. 또한, 병렬 처리를 필요로 하는 작업의 경우 Python의 멀티프로세싱 모듈이나 써드파티 라이브러리를 사용하여 이 문제를 해결할 수 있다.

GIL을 사용하지 않게 된다면?

GIL을 사용하지 않게 되면, 여러 스레드가 동시에 실행되면서 Python 객체에 접근할 수 있게 됩니다. 이는 멀티 코어 프로세서에서 병렬 작업을 가능하게 하므로, 이론적으로는 프로그램의 실행 속도를 향상시킬 수 있습니다.

그러나 GIL이 없다면, 공유 데이터에 대한 동시 접근을 막기 위한 새로운 방법이 필요하게 됩니다. 즉, 스레드 안전성을 보장하기 위해 더 많은 동기화가 필요해질 수 있습니다. 이렇게 되면 복잡성이 증가하고, 잘못 관리된 경우 데이터의 일관성이 손상될 수 있습니다.

또한, 동기화를 위한 오버헤드로 인해 실제로 성능 향상을 얻지 못하고, 오히려 성능이 저하될 수도 있습니다. 이는 특히 스레드간에 많은 공유 데이터가 있는 경우에 해당됩니다.

따라서 GIL을 제거하는 것은 간단하지 않으며, 잠재적으로 많은 문제를 초래할 수 있습니다. 이런 이유로, Python 커뮤니티는 여러 번 GIL을 제거하려는 시도를 했지만 아직까지는 완전히 성공하지 못했습니다.

django에서 GIL이 없는 python을 쓴다면 어떤 장단점이 있을까?

장점:

  1. 병렬 처리 개선: GIL이 없는 경우, 멀티스레드 환경에서 각 스레드가 독립적으로 동시에 작업을 처리할 수 있습니다. 이로 인해 병렬 처리 성능이 향상될 수 있습니다.

  2. CPU 활용도 증가: GIL이 없다면, 멀티 코어 프로세서의 모든 코어를 동시에 활용할 수 있습니다. 이는 CPU 리소스를 보다 효율적으로 사용할 수 있다는 것을 의미합니다.

단점:

  1. 동기화 문제: GIL이 없는 환경에서는 스레드 간의 동기화를 보다 신중하게 처리해야 합니다. 공유된 데이터에 대한 동시 접근이 발생하면 데이터의 일관성이 손상될 수 있습니다.

  2. 성능 저하 가능성: 과도한 스레드 동기화로 인한 오버헤드가 발생하면, 오히려 성능 저하를 초래할 수 있습니다.

  3. 디버깅 어려움: 병렬 코드는 일반적으로 디버깅이 어렵습니다. 스레드 간의 상호작용이나 경쟁 조건 등으로 인해 발생하는 문제를 찾고 해결하는 것이 복잡할 수 있습니다.

따라서, Django에서 GIL이 없는 Python을 사용하려면 스레드 동기화와 관련된 복잡성을 잘 이해하고 관리할 줄 알아야 합니다. 또한, CPU 활용도와 병렬 처리 성능 향상이 실제로 필요한 경우에만 고려하는 것이 좋습니다.

django는 지금 스레드를 어떤 식으로 쓰고 있어?

Django는 요청을 처리하기 위해 스레드나 프로세스를 사용하는데, 그 방식은 웹 서버의 설정에 따라 달라집니다.

Django는 기본적으로 동기적으로 동작합니다. 즉, 하나의 요청을 처리하는 동안 다른 요청은 대기하게 됩니다. 이런 동작 방식은 간단하고 직관적이지만, 요청이 많아질 경우 효율성이 떨어질 수 있습니다.

그래서 실제 운영 환경에서는 여러 요청을 동시에 처리할 수 있도록 웹 서버를 설정합니다. 예를 들어, WSGI 서버인 Gunicorn을 사용하는 경우, 워커(worker)라는 개념을 통해 요청을 병렬로 처리할 수 있습니다. 워커는 별도의 프로세스나 스레드로 동작하며, 각 워커는 독립적으로 요청을 처리합니다. 이렇게 함으로써 Gunicorn은 여러 요청을 동시에 처리할 수 있습니다.

Django 3.1 이상에서는 비동기 기능이 추가되어, ASGI(Application Server Gateway Interface)를 사용해 비동기 요청도 처리할 수 있습니다. 이를 사용하면, Django는 웹소켓과 같은 비동기 프로토콜을 지원하며, 비동기 뷰, 미들웨어, 테스트 클라이언트를 지원합니다.

그러나 Python의 GIL 때문에, 싱글 프로세스 내에서의 멀티스레딩은 여전히 CPU-bound 작업에서는 병렬성을 제공하지 못합니다. 따라서 CPU 사용이 많은 작업을 수행하는 Django 애플리케이션의 경우, 여러 프로세스를 사용하는 것이 더 효율적일 수 있습니다.

gunicorn에서 멀티프로세스가 아닌 멀티스레드 모드를 사용하는 것이 이점

Gunicorn에서 멀티스레드 모드를 사용하는 것이 이점이 될 수 있는 상황들이 있습니다. 멀티스레드와 멀티프로세스는 각각의 장단점을 가지고 있고, 그 선택은 애플리케이션의 특성과 요구사항에 따라 달라집니다.

멀티스레드 모드의 주요 이점은 다음과 같습니다:

  1. 메모리 효율성: 멀티스레드 모드에서는, 모든 스레드가 같은 메모리 공간을 공유하므로 메모리 사용량이 줄어듭니다. 반면에, 멀티프로세스 모드에서는 각 프로세스가 독립된 메모리 공간을 가지므로, 비슷한 작업을 수행하더라도 메모리 사용량이 늘어납니다.

  2. 컨텍스트 스위칭: 멀티스레드 모드에서 스레드 간의 컨텍스트 스위칭은 멀티프로세스 모드에서 프로세스 간의 컨텍스트 스위칭보다 오버헤드가 적습니다. 이는 스레드가 메모리를 공유하므로 컨텍스트 정보를 교환하는 데 비용이 덜 들기 때문입니다.

그러나 멀티스레드 모드에서는 Python의 GIL(Global Interpreter Lock)이 병목이 될 수 있습니다. 즉, 한 번에 하나의 스레드만이 실행될 수 있으므로 CPU-bound 작업에서는 멀티스레드가 병렬성을 제공하지 못할 수 있습니다.

따라서 Gunicorn에서 멀티스레드 모드를 사용하는 것이 이점이 되는지는 애플리케이션의 특성과 환경에 따라 다릅니다. I/O-bound 작업이 많은 애플리케이션에서는 멀티스레드가 효과적일 수 있으며, 반대로 CPU-bound 작업이 많은 애플리케이션에서는 멀티프로세스가 더 나을 수 있습니다.

DB데이터를 시리얼라이즈 하는 일이 대부분이라면

데이터베이스에서 데이터를 가져오는 작업은 대부분 I/O-bound 작업입니다. 즉, 이러한 작업은 CPU 연산보다는 네트워크나 디스크 I/O에 의해 성능이 결정됩니다. 따라서 이런 경우, 멀티스레드 모드가 더 적합할 수 있습니다.

멀티스레드 모드에서는 스레드들이 메모리를 공유하기 때문에, 메모리 사용량이 더 적습니다. 그리고 스레드 간의 컨텍스트 스위칭 비용이 프로세스 간의 컨텍스트 스위칭 비용보다 적기 때문에, I/O-bound 작업에서는 더 효율적일 수 있습니다.

하지만 멀티스레드 환경에서는 동시성 제어에 주의해야 합니다. 다수의 스레드가 동일한 데이터에 동시에 접근하는 경우, 데이터의 일관성을 유지하기 위한 동기화가 필요합니다. 이는 프로그래밍의 복잡성을 증가시키고, 잘못 관리되면 버그를 유발할 수 있습니다.

따라서 이러한 요소들을 종합적으로 고려하여, 애플리케이션의 특성, 트래픽 패턴, 서버의 자원 등에 따라 가장 적합한 모드를 선택하는 것이 중요합니다.

스레드를 사용하지 않는 편이 개발 난이도가 낮다고 할 수 있을까?

예, 일반적으로 스레드를 사용하지 않는 동기 프로그래밍이 비동기나 멀티스레드 프로그래밍에 비해 개발 난이도가 낮습니다. 이는 다음과 같은 이유 때문입니다:

  1. 복잡성 감소: 멀티스레드 프로그래밍에서는 스레드간의 상호작용 및 동기화 문제를 처리해야 합니다. 이런 문제는 복잡성을 높이며, 종종 버그를 발생시키는 원인이 됩니다.

  2. 디버깅 용이성: 동기 코드는 일반적으로 디버깅하기 쉽습니다. 코드의 흐름이 명확하고 예측 가능하기 때문에 문제를 찾고 해결하는 것이 더 간단합니다. 반면, 비동기나 멀티스레드 코드는 동시성 문제로 인해 디버깅이 더 어려울 수 있습니다.

  3. 코드 가독성: 동기 코드는 위에서 아래로 순차적으로 실행되므로, 가독성이 높고 이해하기 쉽습니다. 비동기나 멀티스레드 코드는 이벤트 루프, 콜백, 프로미스 등의 개념을 이해해야 하며, 코드 흐름을 추적하기 어려울 수 있습니다.

그러나 이러한 이점에도 불구하고, 대규모 서비스나 I/O 작업이 많은 애플리케이션에서는 비동기나 멀티스레드 프로그래밍을 사용하는 것이 효율적일 수 있습니다. 이를 통해 애플리케이션의 성능을 향상시키고, 서버 리소스를 더 효율적으로 활용할 수 있습니다. 따라서 애플리케이션의 요구 사항에 따라 적절한 접근법을 선택하는 것이 중요합니다.
